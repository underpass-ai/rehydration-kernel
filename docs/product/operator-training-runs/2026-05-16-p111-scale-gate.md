## 2. North Star Check

```text
Operator 0.5B:
  only learns to use KMP.

Strong teacher:
  produces semantics when semantics are needed.

Kernel:
  validates, stores, traverses, proves, and audits memory.
```

This attempt is not trying to teach a small model to reason over MemoryArena
itself. It is a scale gate for the same KMP/MCP operating surface already
validated in the golden read run.

The kernel remains deterministic. The Operator is only expected to choose
bounded read actions, use visible refs/cursors/page state, and stop when enough
evidence has been observed.

## 3. Hypothesis

Main hypothesis:

```text
The current P1.11 MemoryArena corpus can be audited, resumed, exported,
trained, evaluated, de-anonymized, and replayed through real KMP/MCP without
unbounded traversal or hidden state.
```

Success means:

- remote audit can inspect the 221-task run in bounded pages with progress and
  explicit resume offsets;
- every checked remote ref belongs to the expected run namespace;
- no remote inspect page reports missing or errored refs;
- SFT data remains grouped by task, anonymized, leak-clean, and visible-ref
  safe;
- offline policy eval has zero invalid actions and zero unbounded calls before
  any live replay;
- live MCP replay starts with a bounded `--limit 100` smoke and only scales if
  the smoke is clean.

Failure means:

- audit pagination is opaque, non-resumable, or reads more than the requested
  page;
- any checked ref is missing from the deployed kernel;
- any mixed `run_id` or foreign ref appears in the projected corpus;
- the training split leaks benchmark answers, future refs, raw refs, hidden
  tool outcomes, or writer-only labels;
- the model can pass offline eval but fails real MCP replay.

## 13. Known Risks

- The 221-task run was recorded before page metadata existed in historical
  stored navigation rows. Fresh live replay records `page` and
  `partial_result`; old exported trajectories cannot retroactively contain
  those fields.
- A single clean audit page proves resume mechanics, not full-corpus remote
  audit completeness.
- Training on this corpus without the golden/conformance rows would learn a
  dominant MemoryArena navigation lane instead of full KMP operation. The
  measured target capability coverage is 41.67%.
- Training with only 50 golden v4 conformance rows restores 100% capability
  presence, but not enough examples per rare capability. Some critical shapes
  have only 1-5 train examples.
- `write_context_read` rows behave differently from normal reader rows. Mixing
  them is useful, but they need either more examples or a separate profile so
  writer pre-read windows do not inherit generic reader window defaults.
- Trace page continuation is still underrepresented. A single train example
  and a single eval example are not enough for a robust paging claim.
- Reusing a live `run_id` can pollute the deployed kernel with previous attempt
  data and create false future-leak symptoms. Every new live run must use a
  fresh `run_id`.

## 17. Next Steps

1. Split `operator-read` and `write_context_read` gates in the reporting so
   smart-writer pre-read cannot block a read-profile claim or hide its own
   failures.
2. Add more page-aware real trajectories so `trace.page:first` and
   `trace.page:continue` are not mostly synthetic.
3. Build a dedicated smart-writer pre-read dataset with tighter examples for
   `near` vs `trace`, window sizing, limit sizing, and inspect-after-near.
4. Add macro/capability eval gates that fail when any required KMP/MCP shape
   has too few examples or too low accuracy, even if row accuracy is high.
5. Generate a real-live rare read replay set if we want to replay
   `read-rare-v1` shapes against the deployed kernel; synthetic conformance rows
   are correct for contract training, not for live replay.
6. Continue remote audit from `--offset 1300` if we want full corpus evidence.
