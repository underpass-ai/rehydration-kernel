# Operator Training Experiment Process

Status: active process
Date: 2026-05-14

This document defines how Operator training attempts are planned, generated,
audited, run, stopped, and recorded.

The goal is to keep every experiment reproducible and inspectable. A failed run
is still useful if it records exactly what failed and why.

Dataset quality is governed by
[`operator-dataset-quality-contract.md`](operator-dataset-quality-contract.md).
That contract is mandatory before GPU training. Contract coverage alone is not
enough: every dataset must also record diversity, action balance, contrastive
families, and majority-baseline collapse risk.

The architecture boundary for these checks is
[`operator-test-architecture.md`](operator-test-architecture.md): new contract
and evaluation logic must be Rust-first, typed, fail-fast, and independent from
kernel domain/application crates.

## North Star

Keep the split of responsibilities visible at all times:

```text
Operator 0.5B:
  only learns to use KMP.

Strong teacher:
  produces semantics when semantics are needed.

Kernel:
  validates, stores, traverses, proves, and audits memory.
```

This means:

- `operator-read` learns bounded KMP/MCP operation;
- `operator-write` does not author rich relations from scratch;
- writer semantics come from a strong offline teacher, with GPT-5.5 as the
  preferred teacher;
- kernel core never becomes an inference layer;
- every dataset row must make clear whether its label is deterministic,
  benchmark-derived, teacher-derived, or human-curated.

## Document Per Attempt

Every serious training attempt gets a dedicated document under:

```text
docs/product/operator-training-runs/
```

Use this naming pattern:

```text
YYYY-MM-DD-<attempt-id>-<short-scope>.md
```

Examples:

```text
2026-05-14-opread-p111-qwen05.md
2026-05-15-opwrite-gpt55-teacher-v1.md
2026-05-16-opread-conformance-expanded-v2.md
```

Copy the template from:

```text
docs/product/operator-training-run-template.md
```

The attempt document is the source of truth for that run. Do not overwrite it
with a later story. If the run fails, mark it failed and create a new attempt
document for the next run.

## Synthetic Data Rules

Synthetic conformance data must be deterministic and honest:

- no LLM call is allowed in the deterministic conformance trajectory builder;
- `target_action` must be generated by explicit synthetic builder logic;
- every target action must pass the shared KMP/MCP action validator before
  export;
- model-facing prompts must include `goal`;
- model-facing prompts must not include `target_action`, benchmark answers,
  hidden writer labels, raw prompts, credentials, or post-hoc tool outputs;
- refs may be anonymized, but `about` values must remain a separate identity
  space from node refs;
- if a target action refers to a ref that is not visible, the row must be
  dropped or the export must fail, depending on the attempt policy;
- profile-specific datasets must use explicit mode filters rather than ad hoc
  file editing;
- when mixing real benchmark tasks with synthetic conformance rows, use a group
  key that preserves benchmark task grouping and lets synthetic rows split by
  step id;
- training claims must use capability-aware splits for the declared profile;
- if both train and eval are required to cover a profile, each required
  capability must appear in at least two distinct groups before splitting;
- a capability reported by the SFT preparer must match the shared contract
  coverage semantics; `kernel_trace` pagination only counts as
  `trace.page:first` or `trace.page:continue` when the action carries an
  explicit `page` object;
- write rows may test prepared writes, anti-invention, and fail-fast behavior,
  but they must not train semantic relation authorship unless the relation was
  produced by a validated teacher pipeline.

Synthetic coverage is not optional. Real benchmark traces give realism, but the
synthetic block must deliberately cover every active KMP/MCP case of use for
the profile. A dataset that relies on MemoryArena, LongMemEval, or any other
benchmark to accidentally exercise `trace`, `stop`, page continuation,
dimension scopes, or writer pre-read boundaries is not trainable.

Minimum rule:

```text
each active use-case family has enough deduplicated synthetic train and eval
states before benchmark traces are mixed in.
```

Use the counts in
[`operator-dataset-quality-contract.md`](operator-dataset-quality-contract.md)
as the gate.

## Capability And Sample Value Metrics

Do not judge a dataset only by row count.

Also do not judge it only by contract coverage. A dataset can have 100%
profile coverage, zero leakage, and zero train/eval overlap while still being
too repetitive to teach the policy. The pre-training audit must prove that the
dataset has enough distinct visible states and enough contrastive examples for
the model to learn when to choose each action.

Each training cut must record what capabilities it adds and whether those
capabilities improve the model under the strict policy evaluator. A sample is
useful only if it teaches an observable KMP/MCP decision without leaking the
answer or making the action contract softer.

Minimum capability evidence:

- declared profile: `operator-read`, `operator-full`, `prepared-write`, or
  `writer-teacher`;
- required capability coverage on all rows;
- required capability coverage on train rows;
- required capability coverage on eval rows;
- target action distribution on train rows;
- target action distribution on eval rows;
- unique model-facing rows per target action;
- unique groups per target action;
- maximum duplicate model-row count;
- majority-action baseline accuracy;
- contrastive families per action boundary;
- capabilities that appear in only one group;
- capabilities added by the new dataset versus the previous baseline;
- capabilities still missing after the merge;
- per-capability exact action accuracy after prediction;
- per-capability invalid prediction count;
- per-capability unbounded call count;
- per-capability missing prediction count.

When comparing two datasets, classify each added data block as:

| Classification | Meaning |
| --- | --- |
| `improves` | It raises strict policy metrics for the capabilities it was meant to teach without introducing contract failures. |
| `neutral` | It changes loss or token accuracy but does not materially change strict policy behavior. |
| `regresses` | It lowers strict policy behavior, increases invalid output, or makes tool choice worse. |
| `unsafe` | It leaks gold data, hidden labels, unavailable refs, or unsupported provenance. |
| `unproven` | It has not been isolated enough to know whether it helped. |

This is especially important for mixed datasets. A larger corpus can make
training loss look better while making the Operator worse at a rare capability
such as `kernel_inspect`, trace pagination, dynamic windows, or multi-about
scope selection.

If the dataset makes a trivial majority-action policy competitive, it is not a
serious training cut yet. Fix the dataset before training.

If a dataset block is marked `regresses` or `unsafe`, do not patch it in place.
Preserve the failed attempt, document the reason, and create a new cut.

## Golden Operator Dataset Policy

The Operator should not be trained only by making benchmark traces larger. A
small model can learn a repeated benchmark lane while still failing the KMP/MCP
contract surface. The `2026-05-14` run showed this explicitly: the model solved
MemoryArena-style rows but failed every held-out synthetic conformance row.

For Operator-read, prefer a small golden dataset before large benchmark mixing.

A golden row is valuable when:

- the state visible to the model contains every value needed for the action;
- the correct action is clear without reading hidden metadata;
- the row teaches a specific KMP/MCP capability;
- the row is part of a contrastive family where a similar state requires a
  different action;
- the target action passes the same contract validator used for predictions.

Do not keep rows that require hidden inference. Visibility checks must cover
refs and non-ref cursors:

- target refs;
- temporal `time` cursors;
- temporal `sequence` cursors;
- trace page cursors;
- explicit about/scope lists.

The SFT preparer exposes:

```bash
--require-visible-target-refs
--require-visible-target-cursors
```

`--require-visible-target-cursors` checks temporal `time`, temporal
`sequence`, and trace `page.cursor` values against `visible_state`.

Every SFT preparation writes:

```text
<output>/debug_audit.jsonl
```

This file is a structured debug log for post-run audits. It records sample
status, split, target tool, target argument keys, target capabilities, visible
refs, visible cursors, missing refs, missing cursors, and exact drop reasons.
Keep it with the attempt artifacts.

The first golden cut is versioned as:

```text
opread-golden-v1
```

Attempt document:

```text
docs/product/operator-training-runs/2026-05-15-opread-golden-v1.md
```

Use large benchmark corpora only after the golden contract gate is clean. When
mixing, cap source dominance so a common benchmark pattern cannot hide rare
KMP/MCP failures.

## Writer Teacher Rules

Writer semantic datasets are different from contract datasets.

A writer teacher row must record:

- `label_source = gpt5_5_teacher`;
- exact `teacher_model`;
- `teacher_prompt_version`;
- `schema_version`;
- KMP/MCP tools allowed during context reading;
- refs inspected before deciding;
- candidate refs considered;
- final relation, `why`, evidence, and confidence;
- explicit fallback or escalation reason when no rich relation is justified.

Fail fast if:

- GPT-5.5 is unavailable for a writer-teacher run;
- output is not strict structured JSON;
- a cited ref was not visible or inspected;
- `why` does not cite both endpoints of a rich relation;
- the teacher emits a relation outside the allowed vocabulary;
- the row cannot pass the same action contract used by runtime Operator
  predictions.

No silent fallback to another model is allowed for writer semantic labels.

## Attempt Lifecycle

### 1. Open Attempt Document

Before generating or training anything:

- copy the run template;
- assign attempt id;
- write hypothesis;
- write scope: `operator-read`, `prepared-write`, `writer-teacher`, or mixed;
- record branch, commit, dirty status, and artifact root;
- declare expected stop gates.

### 2. Generate Or Select Datasets

For each dataset:

- record source command;
- record source run id;
- record output paths;
- record row counts;
- record grouped split rule;
- record hashes for large artifacts;
- record whether the data is deterministic, benchmark-derived, teacher-derived,
  or human-curated.

Large artifacts should live outside the repo, normally under:

```text
../rehydration-kernel-artifacts/operator/<attempt-id>/
```

The versioned attempt document in this repo records paths, hashes, summaries,
and conclusions.

### 3. Run Dataset Audits

Minimum audits:

- duplicate `step_id` check;
- duplicate/colliding dimension id check where relevant;
- no-gold/no-leak audit;
- visible target ref audit;
- action contract validation;
- contract coverage report;
- grouped split verification;
- provenance verification.

Stop the attempt if any required audit fails.

### 4. Train

Record:

- base model;
- tokenizer/model revision when available;
- adapter path;
- epochs;
- batch size;
- gradient accumulation;
- max length;
- dtype;
- LoRA rank/alpha/targets;
- hardware;
- job id;
- exact command or manifest;
- train/eval loss history.

Training can continue only while the run is still answering the original
hypothesis. If a data or contract bug appears, stop and document the failed run.

Training loss is not a release metric.

For chat/SFT action data, loss can fall quickly because the model learns stable
formatting, repeated prompt structure, common refs, or frequent tool shapes.
That is useful for monitoring runtime health, but it does not prove the model
can operate KMP. The real gates are strict JSON parsing, exact action schema,
bounded arguments, per-capability policy metrics, and live MCP replay.

### 5. Monitor With Stop Gates

Stop or pause the run when:

- the wrong dataset or wrong model is being used;
- training input contains leaks or unsupported provenance;
- validation loss deteriorates beyond the configured patience;
- generated predictions produce repeated invalid JSON;
- strict policy eval shows invalid predictions;
- any tool call is unbounded;
- live replay shows missing expected refs;
- MCP/gRPC rejects actions that local policy eval accepted;
- cost/time exceeds the budget written in the attempt document.

Do not "rescue" a bad run by patching artifacts in place. Create a new attempt.

### 6. Predict And Evaluate Offline

Required outputs:

- predictions JSONL;
- raw model outputs;
- prediction failure summary;
- strict policy eval;
- per-row policy details;
- baseline comparison, when available.

The policy evaluator must reject near-contract actions. A prediction is valid
only if it matches the exact action schema.

Always write per-row details when a run may be compared later:

```bash
cargo run -p underpass-operator-evaluation-cli --bin underpass_operator_policy_eval -- \
  --model-facing-eval <operator-sft-dir>/eval.jsonl \
  --predictions <operator-predictions-dir>/predictions.jsonl \
  --output <operator-policy-eval>.json \
  --details-output <operator-policy-details>.jsonl
```

Use `--model-facing-eval` when the model saw anonymized refs such as
`about_0001` and `ref_0001`. Do not compare those predictions directly against
raw trajectories. Raw trajectory eval is only valid after explicit
de-anonymization.

Compare a candidate against a previous baseline with:

```bash
python scripts/operator/compare_operator_policy_details.py \
  --baseline-details <baseline-details>.jsonl \
  --candidate-details <candidate-details>.jsonl \
  --output <candidate-vs-baseline-details>.jsonl \
  --summary-output <candidate-vs-baseline-summary>.json \
  --force
```

This comparison is the preferred way to decide whether a dataset block
improved, regressed, or simply changed loss without improving KMP/MCP tool
operation.

### 7. De-Anonymize And Replay

For anonymized datasets:

- run model-facing policy eval against the SFT eval rows;
- map predicted synthetic refs back to raw refs;
- run policy eval on raw refs;
- run live MCP replay first with a small limit;
- run full replay only if the smoke is clean.

Publication candidates require:

- zero invalid predictions;
- zero unbounded tool calls;
- zero missing predictions;
- zero MCP failures;
- zero missing expected refs in replay.

### 8. Close The Attempt

Close every run with one of:

| Status | Meaning |
| --- | --- |
| `promoted` | Can become current baseline or publication candidate. |
| `baseline-only` | Useful comparison, not a release candidate. |
| `internal-only` | Useful for diagnosis, not external claims. |
| `failed` | The run answered a negative result or hit a blocker. |
| `quarantined` | Keep artifacts, but do not train or claim from them. |
| `aborted` | Stopped before producing meaningful results. |

Record follow-up tasks. Never leave a serious run without a final status.

## Minimum Evidence Checklist

Every completed attempt should link or paste:

- dataset summary;
- no-gold audit;
- contract coverage report;
- training command/job;
- training loss summary;
- prediction summary;
- strict policy eval;
- details diff against baseline when applicable;
- de-anonymized eval;
- MCP replay summary;
- decision and reason.

If any evidence is missing, mark the run as non-public.
